# Техническое задание

## Проект: Python SDK для API СУЗ 3.0

## 1. Общая информация

### 1.1. Наименование

Разработка Python SDK для интеграции с API СУЗ 3.0.

### 1.2. Цель

Создать production-ready Python SDK, который упрощает работу с API СУЗ, закрывает инфраструктурные сложности интеграции и предоставляет удобный typed-интерфейс для основных сценариев работы:

* регистрация установки интеграционного решения;
* получение и обновление клиентского токена;
* создание заказа на эмиссию КМ;
* получение статуса заказа;
* получение кодов маркировки;
* отправка отчётов;
* получение статуса обработки отчётов;
* получение квитанций;
* проверка доступности СУЗ.
  Эти сценарии и требования к токену, `omsConnection`, `clientToken`, а также к `X-Signature` описаны в документе API СУЗ 3.0.   

### 1.3. Основание для разработки

Основанием является документ **API СУЗ 3.0**. В документе описаны:

* методы API;
* процесс регистрации установки интеграционного решения;
* получение `omsConnection`;
* получение клиентского токена;
* требования к подписи запросов;
* форматы ответов и ошибок;
* квитирование и статусы обработки.  

---

# 2. Предмет разработки

Разрабатываемый продукт — Python-библиотека, предназначенная для использования в backend-приложениях, интеграционных сервисах, ERP/MES/АСУТП-системах и внутренних сервисах компании.

SDK должен:

* абстрагировать низкоуровневую работу с HTTP;
* управлять токенами;
* поддерживать подпись запросов;
* валидировать входные и выходные данные;
* предоставлять typed-модели и исключения;
* быть расширяемым для новых товарных групп и методов API.

---

# 3. Область применения

SDK предназначен для:

* внутренних интеграций с СУЗ;
* автоматизации заказов КМ;
* отправки отчётности;
* recovery-сценариев и сервисной диагностики;
* использования в микросервисах и CLI-инструментах.

# 4. Функциональные требования

## 4.1. Базовые сценарии MVP

В первой версии SDK должны быть реализованы следующие возможности.

### 4.1.1. Проверка доступности СУЗ

SDK должен предоставлять метод проверки доступности СУЗ и получения версии СУЗ / API. В API СУЗ для этого предусмотрен метод `ping`. 

Пример публичного вызова:

```python
client.health.ping()
```

### 4.1.2. Регистрация установки интеграционного решения

SDK должен предоставлять метод регистрации установки интеграционного решения и получения `omsConnection`.
Документ требует для этого POST-запрос на `/api/v3/integration/connection?omsId={omsId}`, с передачей `X-Signature` и `X-RegistrationKey`; в ответе при успехе возвращается `omsConnection`.  

Пример публичного вызова:

```python
client.integration.register_connection(address="...", name="...")
```

### 4.1.3. Получение клиентского токена

SDK должен уметь получать клиентский токен по зарегистрированному `omsConnection`.

Требования:

* учитывать, что для одной установки доступен только один активный токен;
* при повторном запросе старый токен становится недействительным;
* учитывать TTL токена;
* поддерживать сценарий с True API;
* предусмотреть расширяемость под сценарий ИС МДЛП.  

### 4.1.4. Автоматическое использование токена

SDK должен автоматически добавлять `clientToken` в заголовки запросов.
Для товарной группы «Лекарственные препараты для медицинского применения» должен поддерживаться альтернативный способ авторизации через `Authorization: token ...`, если это требуется сценарием.  

### 4.1.5. Создание заказа на эмиссию КМ

SDK должен предоставлять метод создания заказа на эмиссию КМ.

Пример вызова:

```python
client.orders.create(...)
```

### 4.1.6. Получение статуса заказа

SDK должен предоставлять метод получения статуса заказа по `orderId` и `gtin`.
В API для этого используется `/api/v3/order/status?omsId={omsId}&orderId={orderId}&gtin={gtin}`. 

### 4.1.7. Получение кодов маркировки

SDK должен предоставлять метод получения кодов маркировки по оформленному заказу.

### 4.1.8. Отправка отчёта об использовании / нанесении

SDK должен предоставлять метод отправки отчёта.

Требования:

* поддерживать подпись запроса там, где она требуется;
* обеспечивать стабильную сериализацию JSON перед подписью;
* возвращать typed-результат.
  Документ отдельно указывает использование откреплённой подписи в Base64 и неподдержку attached signature. 

### 4.1.9. Получение статуса обработки отчёта

SDK должен предоставлять метод получения статуса обработки отчёта по `reportId`.
Документ прямо указывает отдельный метод получения статуса обработки отчёта и связывает детали обработки с квитанцией. 

### 4.1.10. Получение квитанции

SDK должен предоставлять метод получения квитанции или деталей обработки документа / отчёта.
Квитирование и структура событий описаны отдельно в разделе 10 документа.  

### 4.1.11. Закрытие заказа

SDK должен предоставлять метод закрытия заказа после завершения получения КМ.

---

## 4.2. Поддержка подписи

SDK должен поддерживать подпись запросов через абстракцию signer.

Требования:

* подпись должна формироваться как detached CMS signature;
* подпись должна передаваться в `X-Signature` в Base64;
* attached signature не должна использоваться;
* слой подписи должен быть заменяемым. 

Нужно реализовать интерфейс подписи:

```python
class BaseSigner(Protocol):
    def sign_bytes(self, payload: bytes) -> str:
        ...
```

SDK не должен быть жёстко привязан к одной реализации CryptoPro.
Должна быть возможность подключать:

* внешний signer;
* оболочку над CLI-инструментом;
* тестовый `NoopSigner`.

---

## 4.3. Управление токеном

SDK должен реализовать `TokenManager`.

### Требования:

* хранение токена в памяти;
* хранение времени истечения;
* предобновление токена до истечения срока;
* запрет лишних повторных запросов токена;
* потокобезопасная логика обновления;
* поддержка явного `authenticate()`.
  Документ указывает, что повторный запрос токена инвалидирует предыдущий, а TTL для токена через True API составляет 10 часов.  

---

## 4.4. Typed API

SDK должен предоставлять объектный интерфейс:

```python
client.health.ping()
client.integration.register_connection(...)
client.auth.authenticate()
client.orders.create(...)
client.orders.get_status(...)
client.orders.get_codes(...)
client.reports.send_utilisation(...)
client.reports.get_status(...)
client.receipts.get(...)
client.orders.close(...)
```

# 5. Нефункциональные требования

## 5.1. Качество кода

Требования:

* Python 3.11+;
* строгая типизация;
* понятная модульная архитектура;
* docstring на английском;
* единый стиль кода;
* минимальный technical debt.

## 5.2. Надёжность

SDK должен:

* корректно обрабатывать сетевые ошибки;
* повторять только безопасные операции по настроенным правилам;
* не скрывать ошибки СУЗ;
* логировать диагностически важные события без утечки секретов.

## 5.3. Расширяемость

Архитектура должна позволять:

* добавлять новые модули API;
* добавлять новые товарные группы;
* добавлять async-версию клиента;
* подключать альтернативные signer-реализации.

## 5.4. Производительность

SDK должен быть пригоден для использования в интеграционных сервисах под рабочей нагрузкой.
Допускается in-memory cache для токена и служебных данных.

## 5.5. Безопасность

SDK не должен:

* логировать токен в открытом виде;
* логировать закрытые ключи;
* сохранять чувствительные данные без явного решения пользователя.

## 6.2. Разделение слоёв

### transport

Отвечает за:

* HTTP-запрос;
* таймауты;
* retries;
* сериализацию/десериализацию;
* преобразование response в ошибки SDK.

### auth

Отвечает за:

* регистрацию подключения;
* получение токена;
* обновление токена;
* подстановку токена в запросы.

### signing

Отвечает за:

* формирование подписи из bytes;
* возврат Base64 detached CMS.

### models

Отвечает за:

* request/response модели;
* enums;
* нормализацию форматов.

### api

Отвечает за:

* пользовательские high-level методы.

# 7. Модели данных

## 7.1. Общие требования

Для входных и выходных данных использовать Pydantic v2.

## 7.2. Подход к моделям

Не пытаться строить одну универсальную модель на все товарные группы.
Нужно закладывать:

* общую базовую модель;
* специализированные модели по сценариям;
* при необходимости отдельные модели под разные `productGroup`.

Причина: документ содержит множество группоспецифичных правил, полей и ошибок, особенно для отчётов и обработки КМ. Это видно даже по различающимся ошибкам обработки отчётов и событиям квитирования.  

---

# 8. Обработка ошибок

## 8.1. Общие требования

SDK должен преобразовывать ошибки API СУЗ в typed exceptions.

Минимальная иерархия:

```python
class SuzError(Exception): ...
class SuzTransportError(SuzError): ...
class SuzTimeoutError(SuzTransportError): ...
class SuzAuthError(SuzError): ...
class SuzTokenExpiredError(SuzAuthError): ...
class SuzSignatureError(SuzError): ...
class SuzValidationError(SuzError): ...
class SuzApiError(SuzError): ...
class SuzRateLimitError(SuzError): ...
```

## 8.2. Содержимое ошибки

Каждое исключение уровня API должно по возможности содержать:

* HTTP status code;
* код ошибки СУЗ;
* текст ошибки;
* исходный response payload.

## 8.3. Специальные случаи

SDK должен отдельно уметь детектировать:

* ошибку подписи;
* ошибку авторизации;
* ошибки валидации payload;
* ошибки обработки отчёта;
* ошибки бизнес-логики по КМ.
  Документ содержит отдельные таблицы кодов ошибок обработки отчётов и кодов маркировки. 

---

# 9. Логирование

SDK должен поддерживать настраиваемое логирование.

Логировать:

* HTTP method;
* URL без секретов;
* correlation id, если появится;
* время выполнения;
* факт обновления токена;
* retry-события.

Не логировать:

* `clientToken` целиком;
* подписи;
* секреты аутентификации;
* приватные ключи.

# 12. Ограничения первой версии

В первую версию **не включать**:

* полное покрытие всех методов API СУЗ;
* полное покрытие всех товарных групп;
* сразу sync и async;
* сложные recovery-flow с полным восстановлением;
* поддержку всех экзотических сценариев агрегации/расформирования.

Первая версия должна закрывать только core-интеграцию и базовый бизнес-flow.

# 18. Риски проекта

Основные риски:

1. Самая сложная часть — не HTTP, а подпись и auth flow.
2. Некоторые payload’ы и сценарии отличаются по товарным группам.
3. Повторный выпуск токена инвалидирует старый, поэтому нельзя делать наивное обновление токена из разных потоков/процессов. 
4. Детали обработки отчётов могут требовать не только статус отчёта, но и работу с квитанциями.

Цель первой версии — сделать **не полный SDK на 726 страниц API**, а **качественное ядро интеграции**, которое:

* надёжно работает с токенами;
* корректно подписывает запросы;
* даёт удобный Python API;
* покрывает основной бизнес-сценарий.
